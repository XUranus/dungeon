# API Overview

The Dungeon Lord backend is built on **FastAPI** and exposes a RESTful JSON API. All endpoints are prefixed with `/api`.

## Base Information

| Item | Details |
|------|---------|
| **Base URL** | `http://localhost:8000` (default) |
| **Protocol** | HTTP / HTTPS (HTTPS recommended for production) |
| **Content Type** | `application/json` for request and response bodies |
| **Streaming** | SSE (`text/event-stream`) for chat endpoints |
| **Authentication** | JWT Bearer Token (HS256) |

## Authentication Model

The system uses **JWT (JSON Web Token)** authentication signed with the HS256 algorithm.

```mermaid
sequenceDiagram
    participant Client as Client
    participant Server as Server

    Client->>Server: POST /api/auth/login {password}
    Server-->>Client: {"token": "eyJhbG..."}
    Note over Client: Store token (localStorage)

    Client->>Server: GET /api/protected-endpoint
    Note right of Client: Authorization: Bearer eyJhbG...
    Server-->>Client: 200 OK / 401 Unauthorized
```

### Endpoint Classification

API endpoints fall into two categories:

- **Public endpoints** -- No authentication required. Accessible by anyone (e.g., Dashboard, public chat, professor index).
- **Admin endpoints** -- Require a valid JWT Bearer Token (e.g., sources management, settings, admin chat, holdings management).

Pass the token in the `Authorization` header:

```
Authorization: Bearer <your-jwt-token>
```

## Common Response Formats

### Success Response

Most endpoints return a JSON object or array. The exact schema varies per endpoint -- consult the individual endpoint documentation.

### Error Response

All errors follow a unified JSON structure:

```json
{
  "detail": "Human-readable error description"
}
```

### HTTP Status Codes

| Status Code | Meaning | Common Causes |
|-------------|---------|---------------|
| `200` | OK | Request processed successfully |
| `400` | Bad Request | Platform not configured, validation failure |
| `401` | Unauthorized | Missing, invalid, or expired token |
| `404` | Not Found | Path or resource ID does not exist |
| `409` | Conflict | A crawl task is already running |
| `422` | Unprocessable Entity | Request body validation error |
| `429` | Too Many Requests | Daily public chat quota exhausted |
| `500` | Internal Server Error | Unhandled exception |
| `503` | Service Unavailable | Admin password not configured |

## SSE Streaming Format

Chat endpoints (`POST /api/dashboard/chat` and `POST /api/chat`) use **Server-Sent Events** to stream LLM-generated text in real time.

### Dashboard Chat Format (Plain Text Stream)

The public dashboard chat uses a simple text-delta SSE format:

```
data: According
data:  to
data:  the
data:  latest
data:  posts
data: ...
data: [DONE]
```

### Admin Chat Format (Vercel AI SDK UI Message Stream)

The admin chat endpoint (`POST /api/chat`) uses the **Vercel AI SDK UI Message Stream** protocol, which supports structured events for text streaming and tool calling:

```
data: {"type":"start","messageId":"msg-abc123"}

data: {"type":"text-start","id":"text-0"}

data: {"type":"text-delta","id":"text-0","delta":"According"}
data: {"type":"text-delta","id":"text-0","delta":" to"}
data: {"type":"text-delta","id":"text-0","delta":" the"}
data: {"type":"text-delta","id":"text-0","delta":" latest"}

data: {"type":"text-end","id":"text-0"}

data: {"type":"finish"}
data: [DONE]
```

### Common SSE Rules

- Every message is prefixed with `data: `
- Messages are separated by `\n\n` (double newline)
- The stream terminates with `data: [DONE]`
- Clients should concatenate text deltas to build the full response

### Response Headers

```
Content-Type: text/event-stream
Cache-Control: no-cache
X-Accel-Buffering: no    # Disable Nginx buffering
```

:::tip SSE Client Libraries
For the admin chat endpoint, use the Vercel AI SDK's `useChat` hook or `readUIMessageStream` utility. For the dashboard chat, a standard `EventSource` or `fetch` + `ReadableStream` parser is sufficient.
:::

## CORS Configuration

In development mode, the backend allows cross-origin requests from:

- `http://localhost:5173` (Vite dev server)
- `http://localhost:3000`

In production, use an Nginx reverse proxy to serve frontend and backend under the same domain, or update the CORS configuration accordingly.
