# Data Flow

This document describes the complete data lifecycle in the Dungeon Lord system -- from content ingestion through platforms like Zsxq and Zhihu, through text preprocessing, semantic chunking, and vectorization, to final RAG-based question answering.

## Ingestion Pipeline

The ingestion pipeline transforms raw platform content into searchable vector embeddings. It runs as a sequential flow orchestrated by `backend/app/services/ingestion.py`.

```mermaid
flowchart TB
    Start["ingest_platform(platform)"] --> CreateTask["Create CrawlTask<br/>status='running'"]
    CreateTask --> IncrCheck{"Full or Incremental?"}
    IncrCheck -->|"Incremental"| QueryLast["Query latest published_at"]
    IncrCheck -->|"Full"| SinceNone["since = None"]

    QueryLast --> Crawl["crawler.crawl_topics(url_token, since)"]
    SinceNone --> Crawl

    Crawl --> SaveTopics["Save Topics to SQLite<br/>(dedup: platform + platform_topic_id)"]
    SaveTopics --> SaveComments["Save Comments to SQLite<br/>(dedup: platform + platform_comment_id)"]
    SaveComments --> EmbedNew["_embed_new_content()"]

    EmbedNew --> Preprocess["Preprocess text<br/>(_preprocess_content)"]
    Preprocess --> ImgDesc["Image description<br/>(_describe_images)"]
    ImgDesc --> Chunk["Chunk text<br/>(split_text_to_chunks)"]
    Chunk --> GetEmb["Batch embed<br/>(get_embeddings)"]
    GetEmb --> ChromaAdd["Write to ChromaDB<br/>(add_documents)"]
    ChromaAdd --> BM25Update["Update BM25 index<br/>(add_to_bm25_index)"]
    BM25Update --> Done["CrawlTask.status = 'done'"]
```

## Query Pipeline

The RAG query pipeline processes user questions through retrieval, fusion, and generation:

```mermaid
flowchart LR
    Q["User Question"] --> Emb["Embed Question"]
    Emb --> Dense["Dense Retrieval<br/>ChromaDB top-20"]
    Emb --> BM25["BM25 Retrieval<br/>top-20"]
    Dense --> RRF["RRF Fusion<br/>dense_w=1.5, bm25_w=1.0"]
    BM25 --> RRF
    RRF --> TopK["Select Top-K<br/>default: 12"]
    TopK --> Ctx["Build Context<br/>with metadata annotations"]
    Ctx --> LLM["LLM Streaming Call<br/>temperature=0.3"]
    LLM --> SSE["SSE Response<br/>data: {text}"]
```

    Ingest->>DB: Update CrawlTask (status='done', counts)
    API-->>FE: {task_id, status: "running"}

    FE->>API: GET /api/sources/crawl/status

## RAG Query Sequence

A complete RAG query from user input to streamed response:

```mermaid
sequenceDiagram
    actor User as User
    participant FE as Frontend
    participant API as FastAPI
    participant RAG as RAG Engine
    participant Emb as EmbeddingService
    participant Vec as ChromaDB
    participant BM25 as BM25 Index
    participant Tools as ToolsService
    participant LLM as OpenAI LLM

    User->>FE: Type question
    FE->>API: POST /api/chat or /api/dashboard/chat
    Note over FE,API: SSE streaming request

    API->>RAG: rag_query_stream(question, history)

    par Parallel Retrieval
        RAG->>Emb: get_embedding(question)
        Emb-->>RAG: question_vector

        RAG->>Vec: query(question_vector, top_k=20)
        Vec-->>RAG: dense_results[]

        RAG->>BM25: bm25_search(question, top_k=20)
        BM25-->>RAG: bm25_results[]
    end

    RAG->>RAG: reciprocal_rank_fusion(dense, bm25)
    Note over RAG: RRF formula: score = sum(w / (k + rank + 1))<br/>dense_weight=1.5, bm25_weight=1.0, k=60

    RAG->>RAG: Build context (with metadata annotations)
    Note over RAG: Format: [platform | type] title (date)<br/>Source URL: url<br/>Content body

    RAG->>LLM: chat.completions.create(stream=True)
    Note over LLM: system: Financial opinion analyst<br/>user: References + question

    opt Tool Calling
        LLM-->>RAG: tool_call: web_search(query)
        RAG->>Tools: web_search(query)
        Tools-->>RAG: search results
        RAG->>LLM: tool result
    end

    loop Streaming Generation
        LLM-->>RAG: chunk.choices[0].delta.content
        RAG-->>API: yield f"data: {text}\n\n"
        API-->>FE: SSE event
        FE-->>User: Real-time display
    end

    LLM-->>RAG: stream end
    RAG-->>API: yield "data: [DONE]\n\n"
    API-->>FE: SSE done
    FE-->>User: Response complete
```

## Incremental vs Full Crawl

The ingestion system supports two crawling modes, controlled by the `full_crawl` parameter:

### Incremental Mode (Default)

Incremental crawling queries the database for the most recent `published_at` timestamp of the target platform, then only fetches content published after that point. This is the default behavior for both manual triggers and scheduled tasks.

```python
if full_crawl:
    since = None
else:
    last_topic_result = await db.execute(
        select(Topic)
        .where(Topic.platform == platform)
        .order_by(Topic.published_at.desc())
        .limit(1)
    )
    last = last_topic_result.scalar_one_or_none()
    since = last.published_at if last else None
```

**Characteristics**:
- Fetches only new content since the last crawl
- Faster execution, lower API usage
- Default mode for scheduled crawls
- Deduplication by `(platform, platform_topic_id)` prevents duplicate entries

### Full Mode

Full crawling fetches all available content from the target platform regardless of existing data.

**Characteristics**:
- Re-fetches all content from the platform
- Useful after configuration changes or data corruption
- Slower execution, higher API usage
- Existing records are updated via upsert (unique constraint on `platform + platform_topic_id`)
- New chunks are added to the vector store; orphaned chunks from deleted topics are cleaned up

### Deduplication Strategy

Both modes use database-level deduplication:

- **Topics**: Unique constraint on `(platform, platform_topic_id)` -- upsert on conflict
- **Comments**: Unique constraint on `(platform, platform_comment_id)` -- upsert on conflict
- **Semantic chunks**: Before re-embedding, existing chunks for a source are deleted from both ChromaDB and the `SemanticChunk` table

:::tip
Use full crawl sparingly. For regular operations, incremental mode is sufficient and significantly reduces load on both the target platform APIs and the local embedding service.
:::
